# OpenClaw 响应时延追踪

日期：2026-03-18

## 结论

当前“汤猴反应慢”的主要瓶颈不在宿主机资源，而在：

1. Discord 同一频道会话持续累积，导致每轮 prompt 上下文越来越大
2. 对需要联网/检索的请求，OpenClaw 会串行调用多次 `web_fetch` / `exec` 工具
3. 当前 `public` 频道允许普通文本直触发，会放大上下文沉积

长记忆和小模型向量化当前不是主要瓶颈：

- `memory-lancedb` 已启用
- 但当前 LanceDB 里还没有真实记忆条目
- 因此暂时没有“召回大量长期记忆”带来的额外负担

## 宿主机资源结论

采样时主机状态：

- `load average`: `0.10 / 0.22 / 0.45`
- 内存：`7.1Gi` 总量，约 `2.3Gi` 空闲，`2.9Gi` 可用
- `openclaw-gateway` CPU 约 `12%`

结论：

- 不是 CPU 打满
- 不是内存不够
- 不是宿主机本地负载导致的“明显卡住”

## 最近真实会话剖面

### 轻量普通对话

日志显示两轮纯聊天请求：

- `durationMs=13950`，约 `14s`
- `durationMs=17300`，约 `17s`

对应上下文诊断：

- `messages=44` / `46`
- `historyTextChars=46728` / `47727`
- `systemPromptChars=28427`

结论：

- 即使只是普通聊天，当前一轮也已经要带上约 `4.7 万字符` 的历史
- 再叠加约 `2.8 万字符` 的系统提示
- 所以简单回复也会落到十几秒

### 联网检索型请求

最近一轮联网型请求：

- 总耗时：`durationMs=108102`，约 `108s`

该轮内部行为：

- 上下文：
  - `messages=48`
  - `historyTextChars=48455`
  - `systemPromptChars=28427`
- 工具调用：
  - `web_fetch` 并发 2 次
  - 后续又追加 2 次 `web_fetch`
  - 然后连续执行 5 次 `exec`

从日志看，这轮慢并不是某一次工具单独超长，而是：

- 先做模型推理决定工具计划
- 多轮抓网页
- 再多轮 `exec`
- 最后再汇总出答复

所以总耗时被串起来放大到约 `1 分 48 秒`

## Discord 实际消息时间

可见最近频道里出现的实际响应速度差异：

- 普通确认类消息：
  - 用户 `08:20:46`
  - bot `08:21:01`
  - 约 `15s`

- 联网查班次类消息：
  - 用户 `08:39:31`
  - bot 首次说明 `08:41:21`
  - 约 `110s`

这和日志里的 `14~17s` 轻量聊天、`~108s` 联网检索型请求基本一致。

## 当前不是主要瓶颈的项

以下项目当前不是主要慢点：

- 宿主机 CPU
- 宿主机内存
- `memory-lancedb` 长期记忆召回
- LanceDB 本地存储
- 小模型 embedding

说明：

- 当前 `ltm stats` 仍是 `Total memories: 0`
- 所以还没有长期记忆召回成本
- embedding 只有在捕获/召回发生时才会明显参与

## 真正的慢点排序

### 1. 同频道单会话历史过长

当前同一 Discord 频道会话已经累积到：

- `messages=48`
- `historyTextChars≈48KB`

这会把每轮普通对话都拉慢到十几秒。

### 2. 工具链过重

对于联网问题，OpenClaw 当前会：

- 多次 `web_fetch`
- 多次 `exec`
- 再回到模型总结

这类任务自然会进到几十秒到百秒级。

### 3. 频道触发策略过宽

当前 `public` 频道是：

- 普通文本直触发
- 不需要 `@`

这会让频道里的自然聊天也持续进入同一个 session，进一步放大历史。

## 优化建议

### A. 优先收紧触发条件

建议恢复为：

- 必须 `@汤猴` 才触发

这样可以立刻减少无关聊天进入同一会话。

### B. 为联网重任务单独设规则

建议区分：

- 普通聊天
- 联网检索 / 数据抓取

避免每次都默认放开整套工具链。

### C. 定期切断频道会话历史

现在已经启用了长记忆：

- `memory-lancedb`
- `autoCapture=true`
- `autoRecall=true`

所以没有必要无限依赖“同一频道一直滚动的超长会话历史”。

可以考虑：

- 新问题新线程
- 或定期重置当前 channel session

### D. 保留 GPT-5.4 Low，但降低不必要工具使用

如果你坚持主模型不变，那么最有效的优化不是换模型，而是：

- 缩短 live context
- 减少自动工具规划次数

## 当前判断

如果不改触发和会话策略：

- 普通聊天继续会是 `10~20s` 级
- 联网检索继续可能是 `1~2 分钟` 级

如果先把频道恢复成 `@` 触发，并减少会话持续滚动：

- 普通聊天会明显回落
- 联网任务仍然会慢，但会比现在更可控
