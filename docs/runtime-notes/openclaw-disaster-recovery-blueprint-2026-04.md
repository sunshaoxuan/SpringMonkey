# OpenClaw Disaster Recovery Blueprint

Date: 2026-04-22 (Asia/Tokyo)

## Goal

If the host suffers an irreversible failure, this repository should support rebuilding `汤猴` quickly and consistently.

This document is not just about "how to reinstall OpenClaw".

It defines:

- what must survive in Git
- what must be backed up from the host
- what can be regenerated from repo scripts
- what must be verified before calling the rebuilt host usable

## Recovery Target

The recovery target is:

- service boots cleanly
- current runtime guards are reapplied from repo
- LINE and Discord channels are usable
- compaction and qwen timeout policy are restored
- Agent Society runtime and kernel are restored
- news cron jobs and news pipeline can run again
- durable host state that is not in Git can be restored from backup

## Recovery Sources

### Source A: Repository

The repository must contain:

- deployment scripts
- runtime patch installers
- startup guard installers
- current environment baseline docs
- acceptance and drift-control docs
- disaster recovery docs

The repo is the recovery source for behavior.

### Source B: Host Recovery Bundle

The recovery bundle must contain the non-repo host state that would otherwise be lost.

The bundle is the recovery source for machine state.

### Source C: Secrets Store

Secrets are not stored in Git.

Recovery still depends on an external secret source for:

- channel tokens
- API keys
- passwords
- SSH credentials

This blueprint assumes secrets are restored from a separate secure system.

## Host State Classes

### Class 1: Rebuild From Repo

These should be regenerated by pulling the repo and rerunning repo-backed installers:

- current runtime patch scripts
- current startup guard scripts
- current workspace bridge files that are generated by installers
- current browser / memory / agent-society deployment logic

### Class 2: Backup And Restore

These should be backed up because they are host state and not fully represented by the repo:

- `/var/lib/openclaw/.openclaw/openclaw.json`
- `/var/lib/openclaw/.openclaw/cron/jobs.json`
- `/var/lib/openclaw/.openclaw/workspace/`
- `/var/lib/openclaw/.openclaw/state/`
- `/var/lib/openclaw/.openclaw/agents/main/sessions/`
- `/var/lib/openclaw/.openclaw/memory/lancedb/`
- `/etc/systemd/system/openclaw.service.d/`
- `/etc/openclaw/openclaw.env`
- `/usr/local/lib/openclaw/`

### Class 3: Backup Metadata Only

These should not be blindly restored as primary truth, but metadata about them should be preserved:

- `/usr/lib/node_modules/openclaw/dist/`

Reason:

- dist bundles are upstream package output and are expected to change
- local patches should be regenerated from repo installers
- what matters for recovery is knowing what patch families were active, not restoring an old opaque dist tree forever

## Recovery Bundle Content

The recommended recovery bundle should contain:

### Required

- host manifest
- current service drop-ins
- current `openclaw.json`
- current `cron/jobs.json`
- workspace tree
- state tree
- agent session files
- memory database directory
- local guard scripts under `/usr/local/lib/openclaw/`

### Optional But Recommended

- current `dist` manifest:
  - bundle filenames
  - marker presence
  - package version hints
- current journal excerpts for major subsystems:
  - LINE
  - Discord
  - news
  - memory-lancedb
- current host repo HEAD and dirty state

### Excluded From Git But Not From Secure Backup

- `/var/lib/openclaw/.openclaw/secrets/`
- external password and token stores

These should be backed up in a secure system, but not committed into this repository.

## Scheduled Backup Policy

The host should not depend on ad hoc human-triggered backups.

The target operating mode is:

- create one recovery bundle every day on the host
- prune non-key historical bundles automatically
- keep a rolling mix of near-term and long-term checkpoints

Current retention policy:

- keep the latest `7` daily bundles
- keep the latest `8` weekly anchors
- keep the latest `6` monthly anchors

This is intended to balance:

- quick rollback for recent breakage
- enough older anchors for delayed discovery of corruption
- bounded disk growth

The scheduled backup installer for this policy is:

- `scripts/remote_install_openclaw_recovery_timer.py`

## Minimum Rebuild Order

### 1. Recover Host Access

Restore:

- SSH access path
- package manager and network reachability
- `systemd`

### 2. Restore Repo And Pull Current Code

Restore repo checkout under:

- `/var/lib/openclaw/repos/SpringMonkey`

Then:

- `git pull`
- verify expected revision

### 3. Restore Host State Bundle

Restore the latest recovery bundle for:

- config
- cron
- workspace
- state
- sessions
- memory
- service drop-ins
- local guard scripts

Do not yet trust runtime behavior just because files are present.

### 4. Reapply Repo-Backed Installers

Always rerun repo installers after restoring state:

- shared capability baseline
- preemptive compaction guard
- qwen timeout retry policy
- LINE direct visibility watchdog
- agent society runtime guard
- agent society startup guard
- agent society kernel
- memory-lancedb guard

This is the step that converts restored files into current runtime behavior.

### 5. Restore Secrets

Restore channel and service secrets from the secure store:

- LINE
- Discord
- search keys
- any site credentials needed by durable workflows

### 6. Validate Against Current Baseline

Use:

- `openclaw-current-environment-baseline-2026-04.md`
- `OPENCLAW_RELEASE_ACCEPTANCE_AND_DRIFT_CONTROL.md`
- `openclaw-failure-layer-model-2026-04.md`

### 7. Run Domain Verification

At minimum:

- one LINE push smoke test
- one LINE direct visible reply test
- one `openclaw cron run` test
- one Discord/news verification path

## Why Previous Recoveries Regressed

The recent failure pattern was not just "OpenClaw is unstable".

The deeper issue was:

- repo scripts existed
- host state existed
- but the bridge between them was incomplete

Typical examples:

- installer targeted stale bundle shape
- host bundle drifted after upstream update
- config baseline was corrected in one path but not another
- push transport was healthy while agent reply chain still failed

This blueprint fixes that by requiring both:

- state bundle recovery
- repo-driven behavior replay

If a future disaster destroys everything except:

- this repository
- a secure secret store
